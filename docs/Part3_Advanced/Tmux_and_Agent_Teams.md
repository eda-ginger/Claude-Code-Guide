# tmux 환경 설정과 에이전트 팀 조율

이 문서에서는 **tmux를 활용한 Claude Code 실행 환경 세팅**과 **여러 Claude Code 인스턴스를 팀으로 조율하는 에이전트 팀 기능**을 안내합니다.

---

## Part 1: tmux + Claude Code 환경 세팅

### 왜 tmux와 함께 쓰나요?

tmux는 터미널 화면을 분할하고 세션을 유지해주는 도구입니다.

- **화면 분할**: 한 터미널에서 여러 작업을 나란히 볼 수 있음
- **세션 유지**: 터미널을 닫아도 작업이 계속 실행됨
- **복귀 가능**: 나중에 돌아와서 이어서 작업 가능
- **원격 서버에서 작업할 때 인터넷 연결이 끊겨도 세션 유지**

Claude Code처럼 장시간 돌아가는 AI 작업에 딱 맞습니다.

### 시작 전에 알아둘 것

반복적으로 나오는 개념 두 가지만 짚고 갑니다.

- **nano**: 터미널에서 쓰는 메모장입니다. `nano 파일명`을 입력하면 그 파일을 메모장처럼 열어서 편집할 수 있습니다.
- **.zshrc**: macOS 터미널(zsh)이 실행될 때마다 자동으로 읽는 설정 파일입니다. 여기에 단축 명령어를 등록해두면 터미널을 열 때마다 자동으로 적용됩니다.

즉, `nano ~/.zshrc`는 **"터미널 설정 파일을 메모장으로 열기"**라는 뜻입니다.

### Alias란?

아이폰의 "단축어"와 같은 개념입니다. 긴 명령어에 짧은 별명을 붙여서, 그 별명만 입력하면 전체 명령이 실행됩니다.

- **Before**: `tmux has-session -t claude 2>/dev/null || tmux new-session -d -s claude "claude"; tmux attach -t claude`
- **After**: `cl`

### 설정 방법 (macOS 기준)

전체 흐름: **tmux 설치 → 설정 파일 열기 → alias 한 줄 추가 → 저장 → 적용**. 그게 다입니다.

아직 tmux를 설치하지 않았다면 1️⃣부터, 이미 설치되어 있다면 2️⃣부터 시작하세요.

> 설정 파일(.zshrc)을 열면 기존 설정들이 쭉 적혀 있을 수 있습니다. 기존 내용은 건드리지 말고, 맨 아래까지 스크롤해서(`Ctrl+V`를 여러 번 누르면 빠르게 내려갑니다) alias를 추가합니다.

**1️⃣ tmux 설치**
```bash
brew install tmux
```

**1.5️⃣ tmux 마우스 스크롤 활성화**

tmux 기본 설정에서는 마우스 스크롤이 안 됩니다. 아래 설정을 추가하면 마우스 휠로 스크롤, 창 크기 조절, 창 선택 등이 가능해집니다.

```bash
echo 'set -g mouse on' >> ~/.tmux.conf
```

이미 tmux가 실행 중이라면 설정을 즉시 반영합니다:
```bash
tmux source-file ~/.tmux.conf
```

**2️⃣ 설정 파일을 메모장으로 열기**
```bash
nano ~/.zshrc
```

**3️⃣ 맨 아래에 이 한 줄 추가**
```bash
# Claude Code with tmux
alias cl='if [ -z "$TMUX" ]; then tmux has-session -t claude 2>/dev/null || tmux new-session -d -s claude "claude"; tmux attach -t claude; else claude; fi'
```

**4️⃣ 저장하고 나가기**
```
Ctrl + X → Y → Enter
```

**5️⃣ 설정 적용**
```bash
source ~/.zshrc
```

**6️⃣ 끝! 이제 이것만 치면 됩니다**
```bash
cl
```

### 이 alias가 정확히 뭘 하는 건가요

- **일반 터미널에서 `cl` 입력** → tmux 세션이 이미 있으면 그대로 연결, 없으면 새로 만들어서 Claude Code 실행
- **이미 tmux 안에서 `cl` 입력** → 중첩 없이 바로 Claude Code 실행

외울 필요 없습니다. 그냥 `cl`만 치면 상황에 맞게 알아서 처리합니다.

### 기억해야 할 명령어는 세 개뿐

| 명령어 | 기능 |
|--------|------|
| `Ctrl+b` 후 `d` | 세션에서 나가기 (작업은 백그라운드에서 계속 돌아감) |
| `cl` | 다시 돌아오기 |
| `exit` | 완전 종료 |

### 실제 사용 예시

1. `cl`로 Claude Code 시작
2. 복잡한 리팩토링 작업을 맡겨놓고
3. `Ctrl+b`, `d`로 세션에서 나가기
4. 회의 다녀오고
5. `cl`로 돌아와서 결과 확인

> ⚠️ **주의:** alias 이름을 `cc`로 지으면 C 컴파일러 명령어와 충돌할 수 있습니다. `cl`, `ccode`, `claude-tmux` 등 겹치지 않는 이름을 추천합니다.

## Part 2: 에이전트 팀 조율하기

> ⚠️ **실험적 기능**: 에이전트 팀은 기본적으로 비활성화되어 있으며, Claude Code v2.1.32 이상이 필요합니다.

에이전트 팀을 사용하면 **함께 작동하는 여러 Claude Code 인스턴스**를 조율할 수 있습니다. 한 세션이 팀 리더 역할을 하여 작업을 조율하고, 작업을 할당하며, 결과를 종합합니다. 팀원들은 독립적으로 작동하며, 서로 직접 통신합니다.

### 에이전트 팀 활성화

`settings.json`에 환경 변수를 추가합니다:

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

### 에이전트 팀을 사용할 때

에이전트 팀은 **병렬 탐색이 실질적인 가치를 더하는 작업**에 가장 효과적입니다:

- **연구 및 검토**: 여러 팀원이 문제의 다양한 측면을 동시에 조사
- **새로운 모듈 또는 기능**: 팀원들이 각각 별도의 부분을 소유하면서 서로 간섭하지 않음
- **경쟁하는 가설로 디버깅**: 다양한 이론을 병렬로 테스트하고 더 빠르게 답에 수렴
- **교차 계층 조율**: 프론트엔드, 백엔드, 테스트에 걸친 변경 사항

> 에이전트 팀은 조율 오버헤드를 추가하고 단일 세션보다 **훨씬 더 많은 토큰**을 사용합니다. 순차적 작업이나 동일 파일 편집이 필요한 경우 단일 세션이나 subagent가 더 효과적입니다.

### Subagent와 에이전트 팀 비교

| | Subagents | 에이전트 팀 |
|:--|:--|:--|
| **컨텍스트** | 자신의 컨텍스트 윈도우; 결과는 호출자에게 반환 | 자신의 컨텍스트 윈도우; 완전히 독립적 |
| **통신** | 메인 에이전트에게만 결과 보고 | 팀원들이 서로 직접 메시지 전송 |
| **조율** | 메인 에이전트가 모든 작업 관리 | 자체 조율을 통한 공유 작업 목록 |
| **최적 용도** | 결과만 중요한 집중된 작업 | 논의와 협업이 필요한 복잡한 작업 |
| **토큰 비용** | 낮음 | 높음 (각 팀원이 별도 인스턴스) |

### 첫 번째 에이전트 팀 시작하기

Claude에게 자연어로 팀 생성을 요청합니다:

```
CLI 도구를 설계하고 있어. 에이전트 팀을 만들어서 다른 각도로 탐색해줘:
한 명은 UX, 한 명은 기술 아키텍처, 한 명은 반대 의견 제시 역할로.
```

Claude는 공유 작업 목록을 가진 팀을 만들고, 각 관점에 대한 팀원들을 생성합니다.

### 표시 모드

| 모드 | 설명 | 요구사항 |
|------|------|----------|
| **In-process** | 모든 팀원이 메인 터미널 내에서 실행. `Shift+Down`으로 팀원 순환 | 없음 (모든 터미널) |
| **분할 창** | 각 팀원이 자신의 창을 가짐. 모든 출력을 한 번에 확인 가능 | tmux 또는 iTerm2 |

기본값은 `"auto"` — tmux 세션 내에서 실행 중이면 분할 창, 아니면 in-process를 사용합니다.

```json
{
  "teammateMode": "in-process"
}
```

또는 실행 시 플래그로 지정:

```bash
claude --teammate-mode in-process
```

> 💡 iTerm2를 사용할 경우, `it2` CLI를 설치하고 **iTerm2 → Settings → General → Magic → Enable Python API**를 활성화해야 합니다.

### 팀원 및 모델 지정

```
4명의 팀원으로 팀을 만들어서 이 모듈들을 병렬로 리팩토링해줘.
각 팀원은 Sonnet 모델을 사용해.
```

### 계획 승인 요구하기

복잡하거나 위험한 작업의 경우, 팀원이 구현 전에 계획을 제출하도록 요구할 수 있습니다:

```
인증 모듈을 리팩토링할 architect 팀원을 생성해줘.
변경하기 전에 계획 승인을 요구해.
```

팀원이 계획을 제출하면 리더가 검토 후 승인/거부합니다. 거부 시 피드백에 따라 수정 후 재제출합니다.

### 팀원과 직접 대화하기

- **In-process 모드**: `Shift+Down`으로 팀원 순환 → 입력하여 메시지 전송. `Ctrl+T`로 작업 목록 전환
- **분할 창 모드**: 팀원의 창을 클릭하여 직접 상호작용

### 작업 할당 및 요청

공유 작업 목록이 팀 전체의 작업을 조율합니다. 작업은 세 가지 상태를 가집니다:

```
대기 중 → 진행 중 → 완료됨
```

- **리더 할당**: 리더에게 어느 작업을 어느 팀원에게 줄지 지시
- **자체 요청**: 팀원이 작업을 마친 후 다음 미할당 작업을 자동으로 선택

### 팀 종료 및 정리

```
# 특정 팀원 종료
researcher 팀원에게 종료하라고 해줘

# 팀 전체 정리
팀을 정리해줘
```

> ⚠️ **항상 리더를 통해 정리합니다.** 팀원이 정리를 실행하면 리소스가 일관성 없는 상태로 남을 수 있습니다.

### Hooks로 품질 게이트 적용

| Hook | 실행 시점 | 용도 |
|------|-----------|------|
| `TeammateIdle` | 팀원이 유휴 상태가 될 때 | 종료 코드 2로 피드백을 보내 계속 작동하게 함 |
| `TaskCompleted` | 작업 완료 표시 시 | 종료 코드 2로 완료를 방지하고 피드백 전송 |

### 아키텍처

| 구성 요소 | 역할 |
|-----------|------|
| **팀 리더** | 팀을 만들고, 팀원들을 생성하며, 작업을 조율하는 메인 세션 |
| **팀원들** | 할당된 작업에서 독립적으로 작동하는 별도 인스턴스 |
| **작업 목록** | 팀원들이 요청하고 완료하는 공유 작업 항목 |
| **메일박스** | 에이전트 간 통신을 위한 메시징 시스템 |

저장 위치:
- 팀 구성: `~/.claude/teams/{team-name}/config.json`
- 작업 목록: `~/.claude/tasks/{team-name}/`

### 사용 사례 예시

#### 병렬 코드 검토

```
PR #142를 리뷰할 에이전트 팀을 만들어줘. 세 명의 검토자:
- 보안 관점 검토
- 성능 영향 확인
- 테스트 커버리지 검증
각자 검토하고 결과를 보고해줘.
```

#### 경쟁하는 가설로 디버깅

```
사용자들이 메시지 하나 보내면 앱이 종료된다고 합니다.
5명의 에이전트 팀원을 생성해서 각각 다른 가설을 조사해줘.
서로 대화하면서 상대방의 이론을 반박하도록 해줘. 과학적 토론처럼.
합의된 결과를 findings 문서에 업데이트해줘.
```

### 모범 사례

1. **충분한 컨텍스트 제공**: 팀원은 리더의 대화 기록을 상속하지 않으므로, 생성 프롬프트에 작업별 세부 사항 포함
2. **적절한 팀 크기**: 대부분 **3~5명**으로 시작. 팀원당 5~6개 작업 유지
3. **파일 충돌 피하기**: 두 팀원이 동일한 파일을 편집하면 덮어쓰기 발생 → 각 팀원이 다른 파일을 소유하도록 분배
4. **연구/검토부터 시작**: 코드 작성 없이 병렬 탐색의 가치를 먼저 경험
5. **팀원 대기 지시**: 리더가 직접 작업을 시작하면 "팀원들이 완료할 때까지 기다려"라고 지시

### 제한 사항

- In-process 팀원과의 세션 재개(`/resume`, `/rewind`) 미지원
- 작업 상태가 지연될 수 있음 (팀원이 완료 표시를 놓치는 경우)
- 세션당 한 팀, 중첩된 팀 불가
- 리더 고정 (승격/이전 불가)
- 분할 창 모드는 VS Code 통합 터미널, Windows Terminal, Ghostty에서 미지원

---
> 💡 **Tip:** 작은 설정 하나지만, 매일 쓰는 도구의 진입 장벽이 확 낮아집니다. 명령어를 떠올리고 타이핑하는 그 몇 초의 마찰이 사라지면, 도구를 꺼내 드는 빈도 자체가 달라집니다.
