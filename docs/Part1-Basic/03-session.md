# 세션 활용법

프로젝트 설정의 기본을 알았다면, 이제 실제 세션에서 알아두면 유용한 조작법과 작업 패턴을 다룹니다.

---

## 1. 세션 관리

### 방향 수정

작업 중 에이전트가 잘못된 방향으로 가고 있다면:

| 단축키 | 기능 |
|--------|------|
| `Esc` | 작업 중간에 중지 (컨텍스트 보존) |
| `Esc + Esc` 또는 `/rewind` | 이전 대화 및 코드 상태 복원 |
| `"Undo that"` | 변경 사항 되돌리기 요청 |
| `/clear` | 컨텍스트 완전 초기화 |

> 같은 문제에 **두 번 이상 수정**했다면, `/clear`하고 더 나은 프롬프트로 새로 시작하세요.

### 대화 재개

세션을 닫았다가 이어서 작업하고 싶을 때:

```bash
claude --continue    # 가장 최근 대화 재개
claude --resume      # 최근 대화에서 선택
```

### Subagent 활용

에이전트에게 조사 작업을 시키면 메인 컨텍스트가 빠르게 차는데, subagent를 쓰면 **별도 컨텍스트에서 조사**하고 결과만 가져옵니다.

```
subagent를 사용하여 인증 시스템이 토큰 새로 고침을 어떻게 처리하는지 조사해줘
```

### Claude에게 인터뷰 요청하기

큰 기능을 구현하기 전, 요구사항을 명확히 하고 싶을 때:

```
[간단한 설명]을 빌드하고 싶어. AskUserQuestion 도구를 사용해서 자세히 인터뷰해줘.
다 다룬 후 SPEC.md에 완전한 사양을 작성해줘.
```


## 2. Plan Mode — 탐색과 실행 분리

`Shift+Tab`으로 Plan Mode와 Normal Mode를 전환할 수 있습니다. Plan Mode에서는 에이전트가 **파일을 수정하지 않고 분석만** 합니다.

```
[Plan Mode]   /src/auth를 읽고 세션 처리 방식을 이해해줘
[Plan Mode]   Google OAuth를 추가하려면 어떤 파일을 변경해야 해? 계획 작성해줘
[Normal Mode] 계획대로 OAuth 흐름을 구현해줘. 테스트도 작성하고 실행해줘
```

**언제 쓰나:**
- 코드를 파악해야 할 때 → Plan Mode로 분석
- 큰 작업 전 계획을 세울 때 → Plan Mode로 설계
- 실제 수정/생성할 때 → Normal Mode로 전환

> 범위가 명확하고 수정이 작은 작업(오타 수정, 로그 추가)에는 Plan Mode가 필요 없습니다.


## 3. 활용 패턴과 흔한 실수

### 유용한 패턴

**탐색 → 이해 → 수정:**
```
> 이 프로젝트의 전체 구조를 알려줘 → 결제 로직 설명해줘 → 재시도 로직 추가해줘
```

**테스트 주도 수정:**
```
> 실패하는 테스트를 보여줘 → 원인 분석 → 수정 → 재실행 확인
```

**비대화형 모드 (일회성 질문):**
```bash
claude -p "이 프로젝트가 무엇을 하는지 설명하세요"
```

### 흔한 실수

| 실수 | 증상 | 해결 |
|------|------|------|
| 한 세션에서 여러 주제를 섞음 | 에이전트가 혼란, 품질 하락 | 주제 전환 시 `/clear` |
| 같은 문제를 반복 수정 | 컨텍스트 오염, 점점 나빠짐 | 두 번 실패 후 `/clear`하고 재시작 |
| 범위 없이 "조사해줘" | 무한 탐색, 시간 낭비 | 구체적 범위 지정 또는 subagent |
| 결과를 검증 안 함 | 그럴듯하지만 엣지 케이스 누락 | 테스트 실행, 스크린샷 확인 요청 |


## 4. 모델 선택과 비용

### 모델 선택

| 모델 | 컨텍스트 | 특징 | 적합한 작업 |
|------|:--------:|------|------------|
| **Opus 4.6** | 200K (1M 확장) | 최고 품질, 깊은 추론 | 복잡한 설계, 팀 리더, 멀티 에이전트 |
| **Sonnet 4.6** | 200K (1M 확장) | 속도와 품질 균형 | 일반 코딩, 구현, 일상 작업 |
| **Haiku 4.5** | 200K | 가장 빠르고 저렴 | 간단한 질문, 테스트, 반복 작업 |

> 모델 전환: `/model` 명령. 빠른 출력이 필요하면 `/fast` 토글 (같은 모델, 출력만 빠르게).

### 비용 절감: 프롬프트 캐싱

이전에 보낸 동일한 내용을 재사용하여 **비용을 최대 90% 절감**합니다. 핵심은 **세션을 쪼개지 않는 것**입니다.

| 방식 | 비용 배수 |
|------|:---------:|
| 한 세션에서 연속 질문 | ~1.6배 |
| 매번 새 세션에서 질문 | ~6.2배 |

| 플랜 | 캐시 유지 시간 |
|------|:--------------:|
| Max | 1시간 |
| Pro | 5분 |

```
✅ 관련 작업은 하나의 세션에서 몰아서 처리
✅ 주제가 완전히 바뀔 때만 /clear
❌ 질문마다 새 세션 열기
❌ 습관적으로 /clear 남발
```

CLAUDE.md 순서도 캐싱에 영향을 줍니다. 안 바뀌는 규칙을 상단에, 자주 바뀌는 내용을 하단에 두면 캐시 히트율이 올라갑니다.


## 5. 선택 설치

필수는 아니지만 설치하면 편리한 도구들입니다.

### 상태 표시줄 (Status Line)

Claude Code 하단에 컨텍스트 사용량, 비용, Git 상태 등을 실시간으로 표시합니다. API 토큰을 소비하지 않습니다.

```
> /statusline 모델 이름과 컨텍스트 사용률을 프로그레스 바로 보여줘
```

이 한 줄이면 자동으로 스크립트 생성 + 설정 완료.

### tmux — 세션 유지 환경

터미널을 닫아도 Claude Code 세션이 백그라운드에서 유지되고, 언제든 돌아올 수 있습니다.

```bash
# 설치: macOS
brew install tmux
# 설치: Ubuntu / WSL
sudo apt install tmux

# 마우스 스크롤 활성화
echo 'set -g mouse on' >> ~/.tmux.conf
```

### 에이전트 팀 활성화

에이전트 팀(멀티 에이전트 병렬 작업)은 실험적 기능으로 기본 비활성화되어 있습니다. 프로젝트의 `.claude/settings.json`에 추가하세요:

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

tmux 안에서 Claude Code를 실행하면, 에이전트 팀 생성 시 팀원별 분할 창이 자동 생성됩니다.

**`cl` alias** — `~/.zshrc` 또는 `~/.bashrc`에 추가:

```bash
cl() {
  local sess="cl-$(basename "$PWD" | tr -cd 'a-zA-Z0-9_-')"
  if [ -z "$TMUX" ]; then
    tmux has-session -t "$sess" 2>/dev/null || tmux new-session -d -s "$sess" -c "$PWD" "claude"
    tmux attach -t "$sess"
  else
    claude
  fi
}
```

| 명령어 | 기능 |
|--------|------|
| `cl` | Claude Code 시작 (또는 돌아오기) |
| `Ctrl+b` → `d` | 세션에서 나가기 (백그라운드 유지) |
| `exit` | 완전 종료 |

> 프로젝트 폴더별 세션이 자동 분리됩니다: `~/my-project`에서 `cl` → `cl-my-project` 세션.

---

> 💡 **Tip:** 누적된 수정이 있는 긴 세션보다, 더 나은 프롬프트가 있는 깨끗한 세션이 거의 항상 더 나은 결과를 만듭니다. 막히면 `/clear`부터 하세요.
