# 비용 최적화와 모범 사례

Claude Code를 효과적으로 사용하기 위한 **컨텍스트 관리, 프롬프트 작성법, 비용 절감, 세션 관리** 전략을 한 곳에 정리합니다.

---

## 1. 컨텍스트 윈도우 이해하기

컨텍스트 윈도우는 Claude의 **작업 메모리**입니다. 대화가 진행될수록 토큰이 선형으로 쌓입니다.

```
[턴 1] 사용자 메시지 + 응답
[턴 2] 턴 1 전체 + 사용자 메시지 + 응답
  ...→ 계속 누적되어 결국 한계에 도달
```

### 모델별 크기

| 모델 | 컨텍스트 윈도우 | 확장 (베타) |
|------|:---------------:|:-----------:|
| Opus 4.6 | 200K | 1M |
| Sonnet 4.6 | 200K | 1M |
| Haiku 4.5 | 200K | — |

> 200K 초과 시 프리미엄 요금(입력 2배, 출력 1.5배)이 적용됩니다.

### 한계 도달 시 대처

| 상황 | 해결 |
|------|------|
| 대화가 너무 길어짐 | `/compact`로 압축 또는 `/clear` 후 재시작 |
| 큰 파일을 통째로 읽음 | 필요한 부분만 선택적으로 읽기 |
| 반복적인 프로젝트 설명 | CLAUDE.md에 기록하여 자동 로드 |
| 에이전트 팀 사용 시 | 각 팀원이 독립 컨텍스트 → 작업 분배 |

## 2. 프롬프트 작성법

### 구체적으로 요청하기

| 비효율적 | 효과적 |
|----------|--------|
| "이 코드 고쳐줘" | "login.js의 handleSubmit에서 빈 이메일일 때 에러 메시지를 표시해줘" |
| "테스트 만들어줘" | "UserService.create에 대해 유효/무효 입력 단위 테스트를 작성해줘" |
| "성능 개선해줘" | "getUsers API의 N+1 쿼리 문제를 확인하고 수정해줘" |

### 단계적으로 요청하기

큰 작업은 나눠서 요청하면 품질이 높아집니다.

```
# 1단계: 현황 파악
> 현재 인증 시스템의 구조를 분석해줘

# 2단계: 계획 수립
> OAuth 2.0으로 전환하려면 어떤 파일들을 수정해야 하는지 계획을 세워줘

# 3단계: 구현
> 계획대로 1번 항목부터 구현해줘
```

### 검증 방법을 함께 제공하기

> 이것이 할 수 있는 **가장 영향력 있는 단일 작업**입니다.

| 전략 | 예시 |
|------|------|
| 테스트 기준 제공 | "validateEmail 함수를 작성해줘. `user@example.com`은 true, `invalid`는 false. 구현 후 테스트 실행해줘" |
| 시각적 검증 | "[스크린샷 붙여넣기] 이 디자인을 구현하고 결과 스크린샷을 찍어 비교해줘" |
| 근본 원인 해결 | "빌드가 이 오류로 실패: [오류]. 수정하고 빌드 성공하는지 확인해줘. 오류를 억제하지 마" |

### 풍부한 콘텐츠 제공하기

- **`@`로 파일 참조**: 코드 위치를 설명하는 대신 직접 참조
- **이미지 붙여넣기**: 드래그 앤 드롭 또는 복사/붙여넣기
- **URL 제공**: 문서 및 API 참조 링크 전달
- **데이터 파이프**: `cat error.log | claude`로 내용 직접 전송

## 3. Plan Mode — 탐색 → 계획 → 구현

`Shift+Tab`으로 Plan Mode를 전환하여 탐색을 실행과 분리합니다.

```
[Plan Mode]  /src/auth를 읽고 세션 처리 방식을 이해해줘
[Plan Mode]  Google OAuth를 추가하려면 어떤 파일을 변경해야 해? 계획 작성해줘
[Normal Mode] 계획대로 OAuth 흐름을 구현해줘. 테스트도 작성하고 실행해줘
[Normal Mode] 설명적인 메시지로 커밋하고 PR 열어줘
```

> 범위가 명확하고 수정이 작은 작업(오타 수정, 로그 추가)에는 계획이 필요 없습니다.

## 4. 세션 관리

### 방향 수정

| 단축키 | 기능 |
|--------|------|
| `Esc` | 작업 중간에 중지 (컨텍스트 보존) |
| `Esc + Esc` 또는 `/rewind` | 이전 대화 및 코드 상태 복원 |
| `"Undo that"` | 변경 사항 되돌리기 요청 |
| `/clear` | 컨텍스트 완전 초기화 |

> 같은 문제에 **두 번 이상 수정**했다면, `/clear`하고 더 나은 프롬프트로 새로 시작하세요.

### 대화 재개

```bash
claude --continue    # 가장 최근 대화 재개
claude --resume      # 최근 대화에서 선택
```

### Subagent 활용

> 컨텍스트가 핵심 제약이므로 subagent는 가장 강력한 도구 중 하나입니다.

```
subagent를 사용하여 인증 시스템이 토큰 새로 고침을 어떻게 처리하는지 조사해줘
```

### Claude에게 인터뷰 요청하기

큰 기능 구현 전, Claude에게 인터뷰를 시키면 요구사항이 명확해집니다:

```
[간단한 설명]을 빌드하고 싶어. AskUserQuestion 도구를 사용해서 자세히 인터뷰해줘.
기술 구현, UI/UX, 엣지 케이스, 트레이드오프에 대해 질문해.
다 다룬 후 SPEC.md에 완전한 사양을 작성해줘.
```

## 5. 세션 인수인계

### 3가지 기억 저장소

| 저장소 | 역할 | 지속성 |
|--------|------|--------|
| **컨텍스트** | 배경 정보 ("이 프로젝트는 Next.js") | CLAUDE.md에 영구 저장 |
| **메모리** | 영구 메모 ("테스트 없이 커밋 금지") | 세션 간 자동 유지 |
| **핸드오프** | 인수인계서 ("여기까지 했고, 다음엔 이것부터") | 수동 또는 /pack으로 전달 |

### /pack — 자동 세션 정리

세션 종료 전 `/pack`을 실행하면, 3개 AI 에이전트가 병렬로 작업 내역을 분석합니다:

```
Phase 1: 데이터 수집 (메인 Claude)
            │
Phase 2: 분석 (2개 에이전트 병렬)
┌──────────────────┬──────────────────┐
│ memory-extractor │ followup-planner │
│ 영구 지식 추출   │ 인수인계 판단    │
└──────────────────┴──────────────────┘
            │
Phase 3: 검증 → Phase 4: 제안 → Phase 5: 적용 → Phase 6: 리포트
```

### 수동 인수인계

```
> 오늘 작업한 내용을 정리해줘:
> 1. 완료한 작업  2. 진행 중인 작업  3. 다음에 할 작업  4. 주의사항
```

## 6. 비용 최적화

### 프롬프트 캐싱

이전에 보낸 동일한 내용을 재사용하여 **비용을 최대 90% 절감**합니다.

**CLAUDE.md 순서가 비용에 직접 영향:**

```
상단: 프로젝트 규칙, 코딩 컨벤션     ← 거의 안 바뀜 (캐시 히트율 높음)
중단: 아키텍처, 기술 스택 정보        ← 가끔 바뀜
하단: 현재 작업, 진행 상황            ← 자주 바뀜 (캐시 미스 허용)
```

### .claudeignore 설정

```
node_modules/
dist/
build/
package-lock.json
yarn.lock
*.min.js
*.map
```

> 이것만으로 토큰 소비가 **60~90% 줄어듭니다.**

### 세션 쪼개지 않기

캐시는 **세션 내에서만 유효**합니다:

| 방식 | 비용 배수 |
|------|:---------:|
| 한 세션에서 연속 질문 | ~1.6배 |
| 매번 새 세션에서 질문 | ~6.2배 |

| 플랜 | 캐시 유지 시간 |
|------|:--------------:|
| Max | 1시간 |
| Pro | 5분 |

```
✅ 관련 작업은 하나의 세션에서 몰아서 처리
✅ 주제가 완전히 바뀔 때만 /clear
❌ 질문마다 새 세션 열기
❌ 습관적으로 /clear 남발
```

## 7. CLAUDE.md 작성법

> `/init`을 실행하여 시작 CLAUDE.md를 생성한 다음, 시간이 지남에 따라 개선하세요.

| 포함할 것 | 제외할 것 |
|-----------|-----------|
| Claude가 추측할 수 없는 Bash 명령 | 코드를 읽어 파악할 수 있는 것 |
| 기본값과 다른 코드 스타일 규칙 | 표준 언어 규칙 |
| 테스트 지시사항, 선호 러너 | 상세한 API 문서 (대신 링크) |
| 저장소 에티켓 (분기 이름, PR 규칙) | 자주 변경되는 정보 |

> 각 줄에 대해 물어보세요: *"이것을 제거하면 Claude가 실수를 할까?"* 아니면 삭제.

| 위치 | 적용 범위 |
|------|-----------|
| `~/.claude/CLAUDE.md` | 모든 프로젝트에 공통 적용 |
| `./CLAUDE.md` | 해당 프로젝트 전체 |
| `./하위디렉토리/CLAUDE.md` | 해당 디렉토리 작업 시 추가 적용 |

## 8. 활용 패턴

### 탐색 → 이해 → 수정

```
> 이 프로젝트의 전체 구조와 주요 진입점을 알려줘
> 결제 처리 로직이 어떻게 동작하는지 설명해줘
> 결제 실패 시 재시도 로직을 추가해줘
```

### 테스트 주도 수정

```
> 현재 실패하는 테스트를 보여줘 → 원인 분석 → 수정 → 재실행 확인
```

### 자동화 — 비대화형 모드

```bash
claude -p "이 프로젝트가 무엇을 하는지 설명하세요"
claude -p "모든 API 엔드포인트 나열" --output-format json
```

### 병렬 실행 — Writer/Reviewer 패턴

| 세션 A (작성자) | 세션 B (검토자) |
|----------------|----------------|
| `속도 제한기 구현` | |
| | `rateLimiter.ts를 검토. 엣지 케이스, 경쟁 조건 찾기` |
| `검토 피드백 반영: [세션 B 출력]` | |

## 9. 흔한 실패 패턴

| 패턴 | 증상 | 해결 |
|------|------|------|
| **주방 싱크 세션** | 한 세션에서 관련 없는 작업을 섞음 | 관련 없는 작업 간에 `/clear` |
| **반복적 수정** | 수정하고, 또 잘못하고... | 두 번 실패 후 `/clear`하고 재시작 |
| **과도한 CLAUDE.md** | 너무 길어서 중요 규칙 무시 | Claude가 이미 올바르게 하는 것은 삭제 |
| **신뢰-검증 간격** | 그럴듯하지만 엣지 케이스 미처리 | 항상 검증(테스트, 스크린샷) 제공 |
| **무한 탐색** | 범위 없이 "조사하세요" 요청 | 좁게 범위 지정 또는 subagent 사용 |

## 10. 모델 선택과 트러블슈팅

### 모델 선택

| 모델 | 특징 | 전환 |
|------|------|------|
| **Opus** | 최고 품질, 복잡한 추론 | `/model` 명령 |
| **Sonnet** | 속도와 품질 균형, 일반 작업 | `/model` 명령 |
| **Haiku** | 가장 빠르고 저렴, 간단한 작업 | `/model` 명령 |

### 트러블슈팅

| 문제 | 해결 |
|------|------|
| Permission denied | `claude --allowedTools Bash,Read,Edit` |
| 컨텍스트 초과 | `/compact` 압축 또는 `/clear` 후 필요한 맥락만 전달 |
| 업데이트 | `npm update -g @anthropic-ai/claude-code` |
| 네트워크 문제 | `export HTTPS_PROXY=http://proxy.company.com:8080` |
