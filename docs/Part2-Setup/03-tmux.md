# tmux 환경 설정과 에이전트 팀 조율

이 문서에서는 **tmux를 활용한 Claude Code 실행 환경 세팅**과 **여러 Claude Code 인스턴스를 팀으로 조율하는 에이전트 팀 기능**을 안내합니다.

---

## Part 1: tmux + Claude Code 환경 세팅

### 왜 tmux와 함께 쓰나요?

tmux는 터미널 화면을 분할하고 세션을 유지해주는 도구입니다.

- **화면 분할**: 한 터미널에서 여러 작업을 나란히 볼 수 있음
- **세션 유지**: 터미널을 닫아도 작업이 계속 실행됨
- **복귀 가능**: 나중에 돌아와서 이어서 작업 가능
- **원격 서버에서 작업할 때 인터넷 연결이 끊겨도 세션 유지**

Claude Code처럼 장시간 돌아가는 AI 작업에 딱 맞습니다.

### 시작 전에 알아둘 것

반복적으로 나오는 개념 두 가지만 짚고 갑니다.

- **nano**: 터미널에서 쓰는 메모장입니다. `nano 파일명`을 입력하면 그 파일을 메모장처럼 열어서 편집할 수 있습니다.
- **.zshrc**: macOS 터미널(zsh)이 실행될 때마다 자동으로 읽는 설정 파일입니다. 여기에 단축 명령어를 등록해두면 터미널을 열 때마다 자동으로 적용됩니다.

즉, `nano ~/.zshrc`는 **"터미널 설정 파일을 메모장으로 열기"**라는 뜻입니다.

### Alias란?

아이폰의 "단축어"와 같은 개념입니다. 긴 명령어에 짧은 별명을 붙여서, 그 별명만 입력하면 전체 명령이 실행됩니다.

- **Before**: `tmux has-session -t claude 2>/dev/null || tmux new-session -d -s claude "claude"; tmux attach -t claude`
- **After**: `cl`

### 설정 방법 (macOS 기준)

전체 흐름: **tmux 설치 → 설정 파일 열기 → alias 한 줄 추가 → 저장 → 적용**. 그게 다입니다.

아직 tmux를 설치하지 않았다면 1️⃣부터, 이미 설치되어 있다면 2️⃣부터 시작하세요.

> 설정 파일(.zshrc)을 열면 기존 설정들이 쭉 적혀 있을 수 있습니다. 기존 내용은 건드리지 말고, 맨 아래까지 스크롤해서(`Ctrl+V`를 여러 번 누르면 빠르게 내려갑니다) alias를 추가합니다.

**1️⃣ tmux 설치**
```bash
brew install tmux
```

**1.5️⃣ tmux 마우스 스크롤 활성화**

tmux 기본 설정에서는 마우스 스크롤이 안 됩니다. 아래 설정을 추가하면 마우스 휠로 스크롤, 창 크기 조절, 창 선택 등이 가능해집니다.

```bash
echo 'set -g mouse on' >> ~/.tmux.conf
```

이미 tmux가 실행 중이라면 설정을 즉시 반영합니다:
```bash
tmux source-file ~/.tmux.conf
```

**2️⃣ 설정 파일을 메모장으로 열기**
```bash
nano ~/.zshrc
```

**3️⃣ 맨 아래에 이 한 줄 추가**
```bash
# Claude Code with tmux
alias cl='if [ -z "$TMUX" ]; then tmux has-session -t claude 2>/dev/null || tmux new-session -d -s claude "claude"; tmux attach -t claude; else claude; fi'
```

**4️⃣ 저장하고 나가기**
```
Ctrl + X → Y → Enter
```

**5️⃣ 설정 적용**
```bash
source ~/.zshrc
```

**6️⃣ 끝! 이제 이것만 치면 됩니다**
```bash
cl
```

### 이 alias가 정확히 뭘 하는 건가요

- **일반 터미널에서 `cl` 입력** → tmux 세션이 이미 있으면 그대로 연결, 없으면 새로 만들어서 Claude Code 실행
- **이미 tmux 안에서 `cl` 입력** → 중첩 없이 바로 Claude Code 실행

외울 필요 없습니다. 그냥 `cl`만 치면 상황에 맞게 알아서 처리합니다.

### 기억해야 할 명령어는 세 개뿐

| 명령어 | 기능 |
|--------|------|
| `Ctrl+b` 후 `d` | 세션에서 나가기 (작업은 백그라운드에서 계속 돌아감) |
| `cl` | 다시 돌아오기 |
| `exit` | 완전 종료 |

### 실제 사용 예시

1. `cl`로 Claude Code 시작
2. 복잡한 리팩토링 작업을 맡겨놓고
3. `Ctrl+b`, `d`로 세션에서 나가기
4. 회의 다녀오고
5. `cl`로 돌아와서 결과 확인

> ⚠️ **주의:** alias 이름을 `cc`로 지으면 C 컴파일러 명령어와 충돌할 수 있습니다. `cl`, `ccode`, `claude-tmux` 등 겹치지 않는 이름을 추천합니다.

### 프로젝트별 세션 분리 — 업그레이드 버전

위의 기본 alias는 모든 프로젝트에서 같은 `claude` 세션을 공유합니다. 여러 프로젝트를 동시에 작업할 때는 세션이 섞여서 불편합니다. 아래 함수 버전은 **현재 폴더명 기반으로 세션을 자동 분리**합니다.

```bash
# ~/.zshrc 또는 ~/.bashrc에 추가
cl() {
  local sess="cl-$(basename "$PWD" | tr -cd 'a-zA-Z0-9_-')"
  if [ -z "$TMUX" ]; then
    tmux has-session -t "$sess" 2>/dev/null || tmux new-session -d -s "$sess" -c "$PWD" "claude --dangerously-skip-permissions"
    tmux attach -t "$sess"
  else
    claude --dangerously-skip-permissions
  fi
}
```

이렇게 하면:
- `~/my-project`에서 `cl` → `cl-my-project` 세션 생성
- `~/other-app`에서 `cl` → `cl-other-app` 세션 생성
- 프로젝트마다 독립된 Claude Code 세션이 유지됩니다

### cl-claw — 텔레그램 봇 연동 세션

텔레그램 Channels 플러그인과 함께 Claude Code를 실행하는 명령어입니다. `cl`과 마찬가지로 프로젝트별 세션이 자동 분리되며, 텔레그램에서 봇으로 작업을 지시할 수 있게 됩니다.

```bash
# ~/.zshrc 또는 ~/.bashrc에 추가
clclaw() {
  local sess="claw-$(basename "$PWD" | tr -cd 'a-zA-Z0-9_-')"
  if [ -z "$TMUX" ]; then
    tmux has-session -t "$sess" 2>/dev/null || tmux new-session -d -s "$sess" -c "$PWD" "claude --channels plugin:telegram@claude-plugins-official --dangerously-skip-permissions"
    tmux attach -t "$sess"
  else
    claude --channels plugin:telegram@claude-plugins-official --dangerously-skip-permissions
  fi
}
alias cl-claw='clclaw'
```

#### cl과 cl-claw 비교

| | `cl` | `cl-claw` |
|--|------|-----------|
| **세션 접두사** | `cl-{폴더명}` | `claw-{폴더명}` |
| **텔레그램 연동** | ❌ | ✅ |
| **사용 시나리오** | 로컬 작업, 직접 조작 | 외출 중 원격 지시, 백그라운드 작업 |

#### 프로젝트별 봇 운영 구조

`cl-claw`의 핵심은 **프로젝트마다 전용 텔레그램 봇을 두는 것**입니다.

```
프로젝트 A/.claude/channels/telegram/.env  →  봇 A 토큰
프로젝트 B/.claude/channels/telegram/.env  →  봇 B 토큰
```

- 각 프로젝트 폴더에 `.claude/channels/telegram/.env` 파일로 봇 토큰을 설정
- `cl-claw`를 해당 프로젝트 폴더에서 실행하면 그 프로젝트 전용 봇이 활성화
- 유저 레벨(`~/.claude/channels/telegram/.env`)에는 봇을 두지 않는 것을 권장 — 프로젝트별 봇과 충돌할 수 있음

#### 실제 사용 흐름

```
1. 프로젝트 폴더로 이동
   cd ~/my-project

2. 텔레그램 봇 연동 세션 시작
   cl-claw

3. 터미널에 "Listening for channel messages..." 확인

4. Ctrl+b, d로 세션에서 나가기 (백그라운드 유지)

5. 텔레그램에서 봇에게 작업 지시
   "테스트 실행해줘", "README 업데이트해줘" 등

6. 돌아와서 결과 확인
   cl-claw
```

> 💡 **Tip:** `cl`은 직접 작업할 때, `cl-claw`는 텔레그램으로 원격 지시할 때로 나눠서 쓰면 편합니다. 같은 프로젝트에서 두 세션을 동시에 띄울 수도 있습니다 — 세션 이름이 다르기 때문에 충돌하지 않습니다.

## Part 2: 에이전트 팀 조율하기

> ⚠️ **실험적 기능**: 에이전트 팀은 기본적으로 비활성화되어 있으며, Claude Code v2.1.32 이상이 필요합니다.

에이전트 팀을 사용하면 **함께 작동하는 여러 Claude Code 인스턴스**를 조율할 수 있습니다. 한 세션이 팀 리더 역할을 하여 작업을 조율하고, 작업을 할당하며, 결과를 종합합니다. 팀원들은 독립적으로 작동하며, 서로 직접 통신합니다.

### 에이전트 팀 활성화

`settings.json`에 환경 변수를 추가합니다:

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

### 에이전트 팀을 사용할 때

에이전트 팀은 **병렬 탐색이 실질적인 가치를 더하는 작업**에 가장 효과적입니다:

- **연구 및 검토**: 여러 팀원이 문제의 다양한 측면을 동시에 조사
- **새로운 모듈 또는 기능**: 팀원들이 각각 별도의 부분을 소유하면서 서로 간섭하지 않음
- **경쟁하는 가설로 디버깅**: 다양한 이론을 병렬로 테스트하고 더 빠르게 답에 수렴
- **교차 계층 조율**: 프론트엔드, 백엔드, 테스트에 걸친 변경 사항

> 에이전트 팀은 조율 오버헤드를 추가하고 단일 세션보다 **훨씬 더 많은 토큰**을 사용합니다. 순차적 작업이나 동일 파일 편집이 필요한 경우 단일 세션이나 subagent가 더 효과적입니다.

### Subagent와 에이전트 팀 비교

| | Subagents | 에이전트 팀 |
|:--|:--|:--|
| **컨텍스트** | 자신의 컨텍스트 윈도우; 결과는 호출자에게 반환 | 자신의 컨텍스트 윈도우; 완전히 독립적 |
| **통신** | 메인 에이전트에게만 결과 보고 | 팀원들이 서로 직접 메시지 전송 |
| **조율** | 메인 에이전트가 모든 작업 관리 | 자체 조율을 통한 공유 작업 목록 |
| **최적 용도** | 결과만 중요한 집중된 작업 | 논의와 협업이 필요한 복잡한 작업 |
| **토큰 비용** | 낮음 | 높음 (각 팀원이 별도 인스턴스) |

### 첫 번째 에이전트 팀 시작하기

Claude에게 자연어로 팀 생성을 요청합니다:

```
CLI 도구를 설계하고 있어. 에이전트 팀을 만들어서 다른 각도로 탐색해줘:
한 명은 UX, 한 명은 기술 아키텍처, 한 명은 반대 의견 제시 역할로.
```

Claude는 공유 작업 목록을 가진 팀을 만들고, 각 관점에 대한 팀원들을 생성합니다.

### 표시 모드

| 모드 | 설명 | 요구사항 |
|------|------|----------|
| **In-process** | 모든 팀원이 메인 터미널 내에서 실행. `Shift+Down`으로 팀원 순환 | 없음 (모든 터미널) |
| **분할 창** | 각 팀원이 자신의 창을 가짐. 모든 출력을 한 번에 확인 가능 | tmux 또는 iTerm2 |

기본값은 `"auto"` — tmux 세션 내에서 실행 중이면 분할 창, 아니면 in-process를 사용합니다.

```json
{
  "teammateMode": "in-process"
}
```

또는 실행 시 플래그로 지정:

```bash
claude --teammate-mode in-process
```

> 💡 iTerm2를 사용할 경우, `it2` CLI를 설치하고 **iTerm2 → Settings → General → Magic → Enable Python API**를 활성화해야 합니다.

### 팀원 및 모델 지정

```
4명의 팀원으로 팀을 만들어서 이 모듈들을 병렬로 리팩토링해줘.
각 팀원은 Sonnet 모델을 사용해.
```

### 계획 승인 요구하기

복잡하거나 위험한 작업의 경우, 팀원이 구현 전에 계획을 제출하도록 요구할 수 있습니다:

```
인증 모듈을 리팩토링할 architect 팀원을 생성해줘.
변경하기 전에 계획 승인을 요구해.
```

팀원이 계획을 제출하면 리더가 검토 후 승인/거부합니다. 거부 시 피드백에 따라 수정 후 재제출합니다.

### 팀원과 직접 대화하기

- **In-process 모드**: `Shift+Down`으로 팀원 순환 → 입력하여 메시지 전송. `Ctrl+T`로 작업 목록 전환
- **분할 창 모드**: 팀원의 창을 클릭하여 직접 상호작용

### 작업 할당 및 요청

공유 작업 목록이 팀 전체의 작업을 조율합니다. 작업은 세 가지 상태를 가집니다:

```
대기 중 → 진행 중 → 완료됨
```

- **리더 할당**: 리더에게 어느 작업을 어느 팀원에게 줄지 지시
- **자체 요청**: 팀원이 작업을 마친 후 다음 미할당 작업을 자동으로 선택

### 팀 종료 및 정리

```
# 특정 팀원 종료
researcher 팀원에게 종료하라고 해줘

# 팀 전체 정리
팀을 정리해줘
```

> ⚠️ **항상 리더를 통해 정리합니다.** 팀원이 정리를 실행하면 리소스가 일관성 없는 상태로 남을 수 있습니다.

### Hooks로 품질 게이트 적용

| Hook | 실행 시점 | 용도 |
|------|-----------|------|
| `TeammateIdle` | 팀원이 유휴 상태가 될 때 | 종료 코드 2로 피드백을 보내 계속 작동하게 함 |
| `TaskCompleted` | 작업 완료 표시 시 | 종료 코드 2로 완료를 방지하고 피드백 전송 |

### 아키텍처

| 구성 요소 | 역할 |
|-----------|------|
| **팀 리더** | 팀을 만들고, 팀원들을 생성하며, 작업을 조율하는 메인 세션 |
| **팀원들** | 할당된 작업에서 독립적으로 작동하는 별도 인스턴스 |
| **작업 목록** | 팀원들이 요청하고 완료하는 공유 작업 항목 |
| **메일박스** | 에이전트 간 통신을 위한 메시징 시스템 |

저장 위치:
- 팀 구성: `~/.claude/teams/{team-name}/config.json`
- 작업 목록: `~/.claude/tasks/{team-name}/`

### 사용 사례 예시

#### 병렬 코드 검토

```
PR #142를 리뷰할 에이전트 팀을 만들어줘. 세 명의 검토자:
- 보안 관점 검토
- 성능 영향 확인
- 테스트 커버리지 검증
각자 검토하고 결과를 보고해줘.
```

#### 경쟁하는 가설로 디버깅

```
사용자들이 메시지 하나 보내면 앱이 종료된다고 합니다.
5명의 에이전트 팀원을 생성해서 각각 다른 가설을 조사해줘.
서로 대화하면서 상대방의 이론을 반박하도록 해줘. 과학적 토론처럼.
합의된 결과를 findings 문서에 업데이트해줘.
```

### 모범 사례

1. **충분한 컨텍스트 제공**: 팀원은 리더의 대화 기록을 상속하지 않으므로, 생성 프롬프트에 작업별 세부 사항 포함
2. **적절한 팀 크기**: 대부분 **3~5명**으로 시작. 팀원당 5~6개 작업 유지
3. **파일 충돌 피하기**: 두 팀원이 동일한 파일을 편집하면 덮어쓰기 발생 → 각 팀원이 다른 파일을 소유하도록 분배
4. **연구/검토부터 시작**: 코드 작성 없이 병렬 탐색의 가치를 먼저 경험
5. **팀원 대기 지시**: 리더가 직접 작업을 시작하면 "팀원들이 완료할 때까지 기다려"라고 지시

### 제한 사항

- In-process 팀원과의 세션 재개(`/resume`, `/rewind`) 미지원
- 작업 상태가 지연될 수 있음 (팀원이 완료 표시를 놓치는 경우)
- 세션당 한 팀, 중첩된 팀 불가
- 리더 고정 (승격/이전 불가)
- 분할 창 모드는 VS Code 통합 터미널, Windows Terminal, Ghostty에서 미지원

## Part 3: 실전 노하우 — 대규모 병렬 작업에서 배운 것들

> 이 섹션은 120편의 학술 논문을 에이전트로 분석하는 실제 프로젝트에서 얻은 교훈입니다. 에이전트 팀(TeamCreate)과 Subagent(Agent 도구) 모두 시도하며 실패와 성공을 반복한 경험을 바탕으로 합니다.

### 에이전트 팀 vs Subagent — 언제 뭘 써야 하나

이론적으로 에이전트 팀은 "토론과 협업이 필요한 복잡한 작업"에 적합합니다. 하지만 실전에서는 **대부분의 배치 작업에서 Subagent(Agent 도구 직접 호출)가 더 안정적**이었습니다.

| 상황 | 에이전트 팀 | Subagent (Agent 도구) |
|------|------------|----------------------|
| 팀원 간 토론/피드백 필요 | ✅ 적합 | ❌ 단방향만 가능 |
| 독립적인 배치 작업 (파일 분석 등) | ❌ 오버헤드 큼 | ✅ 더 간단하고 안정적 |
| 좀비 프로세스 관리 | ❌ TeamDelete 실패 빈번 | ✅ 자동 정리 |
| 토큰 소모가 큰 작업 (PDF 읽기) | ❌ rate limit 위험 | ✅ 순차 제어 가능 |

**핵심 판단 기준**: 팀원끼리 서로 대화해야 하는가?
- **Yes** → 에이전트 팀
- **No** → Subagent (Agent 도구)

### 작업 유형별 병렬 한도

모든 작업을 동일한 병렬 수로 돌리면 안 됩니다. **작업당 토큰 소모량**에 따라 병렬 수를 조절해야 합니다.

| 작업 유형 | 토큰 소모 | 권장 병렬 수 | 예시 |
|----------|----------|-------------|------|
| API 호출 (검색, 메타데이터) | 낮음 | 3~5명 | 논문 목록 수집, URL 검증 |
| 코드 생성/편집 | 중간 | 2~3명 | 파일별 리팩토링, 테스트 작성 |
| PDF/문서 분석 | **매우 높음** | **1~2명** | 논문 전문 읽기, 보고서 분석 |

```
실제 실패 기록:
5명 × 5편 PDF 분석 → API rate limit (즉시 실패)
3명 × 3편 PDF 분석 → API rate limit (수분 내 실패)
3명 × 1편 PDF 분석 → API rate limit (간헐적 실패)
2명 × 1편 PDF 분석 → ✅ 성공 (120편 전체 완료, rate limit 0회)
```

### 1작업 = 1에이전트 원칙

**하나의 에이전트에 여러 작업을 순차적으로 맡기지 마세요.** 이전 작업의 컨텍스트가 다음 작업의 판단을 오염시킵니다.

```
❌ 잘못된 방법:
Agent A: 논문 1 분석 → 논문 2 분석 → 논문 3 분석
(논문 1의 분류 기준이 논문 2, 3에 영향)

✅ 올바른 방법:
Agent A: 논문 1 분석 → 완료 → 폐기
Agent B: 논문 2 분석 → 완료 → 폐기
Agent C: 논문 3 분석 → 완료 → 폐기
```

이 원칙은 특히 **분류/판단이 필요한 분석 작업**에서 중요합니다. 단순 코드 생성이라면 덜 민감할 수 있습니다.

### Rate Limit 단계적 축소 프로토콜

병렬 작업 중 rate limit이 발생하면, 무작정 재시도하지 말고 **단계적으로 축소**합니다.

```
Level 0 (시작):     2명 병렬
    ↓ rate limit 발생
Level 1:            1명 순차 (3초 간격)
    ↓ rate limit 재발생
Level 2:            1명 순차 (30초 대기)
    ↓ rate limit 재발생
Level 3:            중단 → 사용자에게 보고
```

> ⚠️ rate limit에 걸린 에이전트는 shutdown 응답도 못 받아서 **좀비 상태**가 됩니다. `tmux kill-pane`으로 강제 종료해야 합니다.

### 좀비 에이전트 처리

에이전트 팀을 사용할 때 가장 골치 아픈 문제입니다. rate limit이나 에러로 멈춘 에이전트가 tmux pane을 점유한 채 응답하지 않습니다.

```bash
# 1. 좀비 확인
tmux list-panes -a -F "#{pane_id} #{pane_current_command}" | grep claude

# 2. 개별 종료
tmux kill-pane -t %{pane_id}

# 3. TeamDelete 실패 시 (에이전트 팀 사용 시)
rm -rf ~/.claude/teams/{team-name}/
# 빈 config 재생성 후 TeamDelete 재시도
```

**Subagent(Agent 도구)를 쓰면 좀비 문제가 거의 없습니다.** 이것이 배치 작업에서 에이전트 팀 대신 Subagent를 권장하는 가장 큰 이유입니다.

### 중간 저장 + 커밋 습관

대규모 배치 작업은 중간에 멈출 수 있습니다. **매 N건 완료마다 git commit**하는 습관을 들이세요.

```
매 10편 완료 → git commit (중간 저장)
매 배치 완료 → 진행 상황 기록
전체 완료   → 최종 커밋
```

이렇게 하면:
- 중간에 세션이 끊겨도 **처음부터 다시 할 필요 없음**
- 어디까지 했는지 **git log로 즉시 파악** 가능
- 문제 발생 시 **특정 배치만 롤백** 가능

### 분산 결과의 일관성 보장 — 추출-검수-확정 파이프라인

여러 에이전트가 독립적으로 분석한 결과는 **반드시 일관성 검수 단계**를 거쳐야 합니다.

```
Phase 1: 분산 분석
┌─ Agent A → result_001.md ─┐
├─ Agent B → result_002.md ─┤ (각각 독립 판단)
├─ Agent C → result_003.md ─┤
└─ ...     → result_120.md ─┘
              │
Phase 2a: 추출 (병렬 가능)
              │
    핵심 데이터만 뽑아서 하나의 파일로 통합
              │
Phase 2b: 검수 (단일 에이전트, 단일 컨텍스트)
              │
    전체를 한 눈에 보면서 일관성 교정
              │
Phase 3: 확정
    최종 산출물 생성
```

**왜 Phase 2b가 단일 에이전트여야 하는가?**

120편을 10명이 나눠 분석하면, 같은 패턴을 두고도 에이전트마다 다르게 분류합니다. 실제로 12건의 분류 코드 불일치가 발생했고, 이를 한 에이전트가 전체를 보면서 교정했습니다. 분산 검수를 했다면 이런 패턴을 잡을 수 없었을 것입니다.
