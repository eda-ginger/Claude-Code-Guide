# Step 12: 컨텍스트 관리와 비용 최적화 실전 가이드

이 문서에서는 Claude Code를 실전에서 사용할 때 반드시 알아야 할 **컨텍스트 윈도우 한계 극복법**과 **프롬프트 캐싱을 활용한 API 비용 절감법**을 안내합니다.

> 💡 컨텍스트 관리의 이론적 배경은 [Step 8: 컨텍스트 윈도우 이해하기](./Step8_Context_Window.md)를 참고하세요. 이 문서는 **실전 적용**에 초점을 맞춥니다.

## Part 1: 컨텍스트 윈도우 한계 극복 — 세션 인수인계 전략

### 문제: AI가 갑자기 멍을 때린다

Claude Code를 30분 이상 쓰다 보면 갑자기 엉뚱한 답을 하거나, 방금 한 작업을 잊어버리는 경우가 있습니다. 이것이 **컨텍스트 윈도우 한계**입니다.

```
대화 시작: 🧠 "완벽하게 이해하고 있어요!"
30분 후:   🧠 "...뭘 하고 있었죠?"
```

기본 해결법인 `/compact`(압축)와 `/clear`(초기화)는 알고 계실 겁니다. 하지만 `/clear`를 치면 이전 작업 내용이 전부 날아갑니다.

### 해결: Claude Code의 3가지 기억 저장소

Claude Code에는 세션이 바뀌어도 기억을 유지할 수 있는 3가지 저장 방식이 있습니다:

| 저장소 | 역할 | 예시 | 지속성 |
|--------|------|------|--------|
| **컨텍스트 (Context)** | 배경 정보 | "나는 백엔드 개발자", "이 프로젝트는 Next.js" | CLAUDE.md에 영구 저장 |
| **메모리 (Memory)** | 영구 메모 | "이 프로젝트는 Python 사용", "테스트 없이 커밋 금지" | 세션 간 자동 유지 |
| **핸드오프 (Handoff)** | 인수인계서 | "오늘 여기까지 했고, 다음에 이것부터 해야 함" | 수동으로 다음 세션에 전달 |

### 자동 기억 기능 활용하기

Claude Code는 최근 **자동 기억(Auto Memory)** 기능을 지원합니다. `/memory`를 입력하면 Claude가 이번 세션에서 배운 것들을 확인할 수 있습니다.

```
> /memory
```

직접 기억을 저장하도록 지시할 수도 있습니다:

```
> 이 프로젝트에서는 항상 TypeScript strict 모드를 사용해야 한다는 걸 기억해줘
```

### pack 스킬 (session-pack) — 자동 세션 정리

`pack`은 세션 종료 전에 한 번만 실행하면, **3개의 전문 AI 에이전트가 병렬로** 이번 세션의 작업 내역을 자동으로 분석하고 정리해주는 커스텀 스킬입니다.

> 출처: [ten-builder/ten-builder](https://github.com/ten-builder/ten-builder) (GitHub 오픈소스)

**실행:**

```
> /pack
```

**6단계 작동 방식:**

```
Phase 1: 데이터 수집 (메인 Claude)
         git 변경사항, MEMORY.md, Context 파일, 기존 Handoff 스캔
                    │
                    ▼
Phase 2: 분석 (2개 에이전트 병렬)
┌─────────────────────────┬─────────────────────────┐
│ memory-extractor        │ followup-planner        │
│ Memory + Context 추출   │ Handoff 필요 여부 판단  │
└─────────────────────────┴─────────────────────────┘
                    │
                    ▼
Phase 3: 검증 (순차)
┌───────────────────────────────────────────────────┐
│ duplicate-checker                                 │
│ Phase 2 결과를 기존 문서와 대조하여 중복/충돌 검증│
└───────────────────────────────────────────────────┘
                    │
                    ▼
Phase 4: 제안 목록 표시 → Phase 5: 사용자 선택 후 적용 → Phase 6: 확인 리포트
```

**3개 에이전트의 역할:**

| 에이전트 | 역할 | 저장 위치 |
|----------|------|-----------|
| **Memory Extractor** | 영구 지식 추출 (도구 설정, 프로젝트 구조, 워크플로 패턴 등) | MEMORY.md + Context 파일 |
| **Followup Planner** | 미완성 작업 파악, 다음 할 일 정리, 인수인계서 작성 | Handoff 문서 |
| **Duplicate Checker** | 위 두 에이전트의 결과를 기존 문서와 대조하여 중복/충돌 검증 | (검증 역할) |

정리가 끝나면 번호가 매겨진 제안 목록을 보여주고, **All(전체 적용) / None(취소) / Pick(번호 선택)** 중 선택할 수 있습니다.

**다음 세션에서 이어서 작업하기:**

`/pack`이 생성한 Handoff 문서(`~/.claude/handoff/HANDOFF-{주제}-{날짜}.md`)를 다음 세션에 붙여넣으면 됩니다:

```
> [핸드오프 문서 붙여넣기]
> 어제 여기까지 했고, 이거 이어서 해줘
```

Claude가 인수인계서를 읽고 바로 맥락을 이해한 채 작업을 시작합니다.

**pack 스킬 사용:**

이 저장소를 clone하면 `.claude/skills/session-pack/` 스킬이 자동으로 포함됩니다. Claude Code에서 `/pack`을 입력하면 바로 사용할 수 있습니다.

> 💡 스킬의 원본 저장소: [ten-builder/ten-builder](https://github.com/ten-builder/ten-builder) — `study-vault`(교재 → 학습 노트 변환)와 `study-quiz`(대화형 퀴즈) 스킬도 함께 제공됩니다.

### 세션 인수인계 패턴 (수동)

`pack` 스킬 없이도 직접 인수인계 문서를 만들어 세션 간 맥락을 전달할 수 있습니다.

**세션 종료 전:**

```
> 오늘 작업한 내용을 정리해줘. 다음 항목을 포함해서:
> 1. 이번 세션에서 완료한 작업
> 2. 현재 진행 중인 작업과 상태
> 3. 다음 세션에서 이어서 해야 할 작업
> 4. 발견된 이슈나 주의사항
```

**다음 세션 시작 시:**

```
> 지난 세션 인수인계:
> - 완료: 로그인 API 구현, DB 스키마 설계
> - 진행 중: 회원가입 API (유효성 검사까지 완료)
> - 다음 할 일: 회원가입 API 이메일 인증 로직 추가
> - 주의: rate limiter 미들웨어 아직 미적용
>
> 회원가입 API 이메일 인증부터 이어서 해줘.
```

Claude가 바로 맥락을 이해하고 이어서 작업합니다.

### CLAUDE.md를 활용한 영구 컨텍스트

매번 반복 설명하는 프로젝트 정보는 `CLAUDE.md`에 기록하면 세션마다 자동으로 로드됩니다:

```markdown
# CLAUDE.md

## 프로젝트 규칙
- 언어: TypeScript (strict 모드)
- 테스트: 모든 PR에 유닛 테스트 필수
- 커밋: Conventional Commits 형식 사용

## 아키텍처
- 프론트엔드: Next.js 15 (App Router)
- 백엔드: Express.js + Prisma ORM
- DB: PostgreSQL

## 현재 진행 상황
- 인증 모듈: 완료
- 결제 모듈: 진행 중 (Stripe 연동)
```

> ⚠️ **주의**: `CLAUDE.md`의 **순서가 중요**합니다. 이 내용은 Part 2의 프롬프트 캐싱과 직접 연관됩니다.

## Part 2: 프롬프트 캐싱 최적화로 API 비용 절감

### 프롬프트 캐싱이란?

Claude Code는 매번 AI에게 **전체 대화 컨텍스트**를 보냅니다. 프롬프트 캐싱은 이전에 보낸 동일한 내용을 재사용하여 **비용을 최대 90%까지 줄이는** 기능입니다.

```
일반 요청:    [전체 컨텍스트] → 100% 비용
캐시 히트:    [캐시된 부분 재사용] + [새 부분만 전송] → 10% 비용
```

좋은 소식은 Claude Code에서 프롬프트 캐싱이 **자동으로 적용**된다는 것입니다. 하지만 잘 쓰는 사람과 그냥 쓰는 사람의 비용 차이가 큽니다.

### 최적화 1: CLAUDE.md 순서 배치

캐싱은 **앞에서부터 일치하는 부분**만 재사용합니다. 따라서 `CLAUDE.md`의 순서가 비용에 직접 영향을 줍니다.

**❌ 나쁜 예 — 자주 바뀌는 내용이 상단:**

```markdown
# CLAUDE.md

## 현재 작업 (매일 바뀜)
- 오늘 할 일: 결제 API 리팩토링

## 프로젝트 규칙 (거의 안 바뀜)
- TypeScript strict 모드 사용
- Conventional Commits 형식
```

→ 상단이 바뀌면 그 아래 전체가 캐시 미스 → 비용 증가

**✅ 좋은 예 — 안 바뀌는 내용이 상단:**

```markdown
# CLAUDE.md

## 프로젝트 규칙 (거의 안 바뀜)
- TypeScript strict 모드 사용
- Conventional Commits 형식

## 아키텍처 (가끔 바뀜)
- Next.js 15 + Express.js + PostgreSQL

## 현재 작업 (자주 바뀜)
- 오늘 할 일: 결제 API 리팩토링
```

→ 상단은 계속 캐시 히트 → 비용 절감

**순서 원칙:**

```
상단: 프로젝트 규칙, 코딩 컨벤션     ← 거의 안 바뀜 (캐시 히트율 높음)
중단: 아키텍처, 기술 스택 정보        ← 가끔 바뀜
하단: 현재 작업, 진행 상황            ← 자주 바뀜 (캐시 미스 허용)
```

### 최적화 2: .claudeignore로 불필요한 파일 차단

Claude Code가 가장 토큰을 낭비하는 구간은 **파일을 읽을 때**입니다. `.gitignore`처럼 `.claudeignore`를 프로젝트 루트에 만들어 AI가 읽을 필요 없는 파일을 차단합니다.

**`.claudeignore` 추천 설정:**

```
# 의존성
node_modules/
vendor/
.venv/
__pycache__/

# 빌드 산출물
dist/
build/
.next/
out/

# 락 파일
package-lock.json
yarn.lock
pnpm-lock.yaml
poetry.lock

# 대용량/바이너리
*.min.js
*.min.css
*.map
*.woff
*.woff2
*.ttf

# 기타
.git/
coverage/
.cache/
```

> 📊 이것만으로도 파일 읽기 시 **토큰 소비가 60~90% 줄어든다**는 테스트 결과가 있습니다.

### 최적화 3: 세션 쪼개지 않기

캐시는 **세션 내에서만 유효**합니다. 세션을 자주 새로 시작하면 캐시가 리셋되어 비용이 급증합니다.

**구체적 수치 비교:**

같은 10만 토큰짜리 파일을 읽고 질문 5개를 할 때:

| 방식 | 비용 배수 |
|------|-----------|
| 한 세션에서 연속 질문 | **~1.6배** |
| 매번 새 세션에서 질문 | **~6.2배** |

→ 세션을 쪼개면 비용이 **약 4배** 폭증합니다.

**캐시 유효 시간:**

| 플랜 | 캐시 유지 시간 |
|------|---------------|
| Max 플랜 | **1시간** |
| Pro 플랜 | **5분** |

**실전 규칙:**

```
✅ 관련된 작업은 하나의 세션에서 몰아서 처리
✅ 주제가 완전히 바뀔 때만 /clear로 리셋
❌ 질문마다 새 세션 열기
❌ 습관적으로 /clear 남발
```

## 실전 체크리스트

세션을 시작하기 전에 이 체크리스트를 확인하세요:

### 세션 시작 전

- [ ] `CLAUDE.md` 순서 확인: 안 바뀌는 규칙이 상단에 있는가?
- [ ] `.claudeignore` 파일이 프로젝트 루트에 있는가?
- [ ] 이전 세션의 인수인계 내용이 준비되어 있는가?

### 세션 진행 중

- [ ] 컨텍스트가 80% 이상 차면 `/compact`로 압축
- [ ] 관련 작업은 같은 세션에서 이어서 진행
- [ ] 중요한 결정 사항은 메모리에 저장 요청

### 세션 종료 전

- [ ] 작업 내용 정리 및 인수인계 문서 작성
- [ ] 다음 세션에서 이어야 할 작업 목록 확인
- [ ] 프로젝트 규칙 변경이 있었다면 `CLAUDE.md` 업데이트

## 요약

| 전략 | 효과 | 난이도 |
|------|------|--------|
| `CLAUDE.md` 순서 최적화 | 캐시 히트율 ~2배 상승 | ⭐ 쉬움 |
| `.claudeignore` 설정 | 토큰 60~90% 절감 | ⭐ 쉬움 |
| 세션 쪼개지 않기 | 비용 ~4배 차이 방지 | ⭐ 쉬움 |
| 세션 인수인계 패턴 | 맥락 손실 없는 작업 연속성 | ⭐⭐ 보통 |
| 자동 기억 + 메모리 활용 | 반복 설명 제거 | ⭐⭐ 보통 |

> 💡 **Tip:** 이 모든 최적화는 한 번 세팅하면 끝입니다. `.claudeignore`를 만들고, `CLAUDE.md` 순서를 정리하고, 세션 인수인계 습관만 들이면 비용은 절반으로, 작업 효율은 두 배로 올라갑니다.
