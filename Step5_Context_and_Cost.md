## Part 1: 컨텍스트 윈도우 이해하기

### 컨텍스트 윈도우란?

"컨텍스트 윈도우"는 Claude가 응답을 생성할 때 참조할 수 있는 **모든 텍스트의 총량**입니다. 응답 자체도 포함됩니다.

쉽게 말해 Claude의 **"작업 메모리"**입니다.

- 컨텍스트 윈도우가 **크면** → 더 복잡하고 긴 대화를 처리할 수 있음
- 컨텍스트 윈도우가 **작으면** → 긴 대화에서 일관성 유지가 어려워짐

> 훈련 데이터는 "장기 기억", 컨텍스트 윈도우는 "단기 작업 메모리"에 해당합니다.

### 토큰이 쌓이는 구조

대화가 진행될수록 토큰이 **선형적으로 축적**됩니다:

```
[턴 1] 사용자 메시지 + Claude 응답
[턴 2] 턴 1 전체 + 사용자 메시지 + Claude 응답
[턴 3] 턴 1~2 전체 + 사용자 메시지 + Claude 응답
  ...
→ 계속 누적되어 결국 한계에 도달
```

### 모델별 컨텍스트 윈도우 크기

| 모델 | 기본 컨텍스트 윈도우 | 확장 컨텍스트 (베타) |
|------|---------------------|---------------------|
| Claude Opus 4.6 | 200K 토큰 | 1M 토큰 |
| Claude Sonnet 4.6 | 200K 토큰 | 1M 토큰 |
| Claude Sonnet 4.5 | 200K 토큰 | 1M 토큰 |
| Claude Haiku 4.5 | 200K 토큰 | - |

> 1M 토큰 컨텍스트는 사용 티어 4 이상 조직에서 베타로 사용 가능하며, 200K 초과 시 프리미엄 요금(입력 2배, 출력 1.5배)이 적용됩니다.

### 확장된 사고(Extended Thinking)와 컨텍스트

Claude는 복잡한 문제를 풀 때 **확장된 사고(thinking)** 블록을 생성합니다.

**핵심 포인트: 이전 사고 블록은 자동으로 제거됩니다.**

```
[턴 1] 입력 → 사고 블록 + 응답 출력
[턴 2] 입력(턴 1의 사고 블록은 제외됨) → 새로운 사고 블록 + 응답 출력
```

- 사고 블록은 **생성 시점에만 한 번** 출력 토큰으로 청구
- **후속 턴에서는 자동으로 제거**되어 컨텍스트 공간을 절약

### 컨텍스트 인식 기능

Claude Sonnet 4.6, Sonnet 4.5, Haiku 4.5는 남은 컨텍스트 공간을 실시간으로 추적합니다.

```xml
<system_warning>Token usage: 35000/200000; 165000 remaining</system_warning>
```

### 컨텍스트 윈도우 한계 도달 시

| 상황 | 해결 방법 |
|------|-----------|
| 대화가 너무 길어짐 | `/compact`로 압축 또는 `/clear` 후 재시작 |
| 큰 파일을 통째로 읽음 | 필요한 부분만 선택적으로 읽기 |
| 반복적인 프로젝트 설명 | CLAUDE.md에 기록하여 자동 로드 |
| 에이전트 팀 사용 시 | 각 팀원이 독립 컨텍스트를 가지므로 작업 분배로 해결 |

## Part 3: 세션 인수인계 전략

### 문제: AI가 갑자기 멍을 때린다

Claude Code를 30분 이상 쓰다 보면 갑자기 엉뚱한 답을 하거나, 방금 한 작업을 잊어버리는 경우가 있습니다. 이것이 **컨텍스트 윈도우 한계**입니다.

```
대화 시작: 🧠 "완벽하게 이해하고 있어요!"
30분 후:   🧠 "...뭘 하고 있었죠?"
```

### Claude Code의 3가지 기억 저장소

| 저장소 | 역할 | 예시 | 지속성 |
|--------|------|------|--------|
| **컨텍스트 (Context)** | 배경 정보 | "나는 백엔드 개발자", "이 프로젝트는 Next.js" | CLAUDE.md에 영구 저장 |
| **메모리 (Memory)** | 영구 메모 | "이 프로젝트는 Python 사용", "테스트 없이 커밋 금지" | 세션 간 자동 유지 |
| **핸드오프 (Handoff)** | 인수인계서 | "오늘 여기까지 했고, 다음에 이것부터 해야 함" | 수동으로 다음 세션에 전달 |

### 자동 기억 기능 활용하기

```
> /memory
> 이 프로젝트에서는 항상 TypeScript strict 모드를 사용해야 한다는 걸 기억해줘
```

### pack 스킬 (session-pack) — 자동 세션 정리

`pack`은 세션 종료 전에 한 번만 실행하면, **3개의 전문 AI 에이전트가 병렬로** 이번 세션의 작업 내역을 자동으로 분석하고 정리해주는 커스텀 스킬입니다.

> 출처: [ten-builder/ten-builder](https://github.com/ten-builder/ten-builder) (GitHub 오픈소스)

**실행:**

```
> /pack
```

**6단계 작동 방식:**

```
Phase 1: 데이터 수집 (메인 Claude)
         git 변경사항, MEMORY.md, Context 파일, 기존 Handoff 스캔
                    │
                    ▼
Phase 2: 분석 (2개 에이전트 병렬)
┌─────────────────────────┬─────────────────────────┐
│ memory-extractor        │ followup-planner        │
│ Memory + Context 추출   │ Handoff 필요 여부 판단  │
└─────────────────────────┴─────────────────────────┘
                    │
                    ▼
Phase 3: 검증 (순차)
┌───────────────────────────────────────────────────┐
│ duplicate-checker                                 │
│ Phase 2 결과를 기존 문서와 대조하여 중복/충돌 검증│
└───────────────────────────────────────────────────┘
                    │
                    ▼
Phase 4: 제안 목록 표시 → Phase 5: 사용자 선택 후 적용 → Phase 6: 확인 리포트
```

**3개 에이전트의 역할:**

| 에이전트 | 역할 | 저장 위치 |
|----------|------|-----------|
| **Memory Extractor** | 영구 지식 추출 | MEMORY.md + Context 파일 |
| **Followup Planner** | 미완성 작업 파악, 인수인계서 작성 | Handoff 문서 |
| **Duplicate Checker** | 중복/충돌 검증 | (검증 역할) |

**pack 스킬 사용:**

이 저장소를 clone하면 `.claude/skills/session-pack/` 스킬이 자동으로 포함됩니다. Claude Code에서 `/pack`을 입력하면 바로 사용할 수 있습니다.

> 💡 스킬의 원본 저장소: [ten-builder/ten-builder](https://github.com/ten-builder/ten-builder) — `study-vault`(교재 → 학습 노트 변환)와 `study-quiz`(대화형 퀴즈) 스킬도 함께 제공됩니다.

### 세션 인수인계 패턴 (수동)

**세션 종료 전:**

```
> 오늘 작업한 내용을 정리해줘. 다음 항목을 포함해서:
> 1. 이번 세션에서 완료한 작업
> 2. 현재 진행 중인 작업과 상태
> 3. 다음 세션에서 이어서 해야 할 작업
> 4. 발견된 이슈나 주의사항
```

**다음 세션 시작 시:**

```
> [핸드오프 문서 붙여넣기]
> 어제 여기까지 했고, 이거 이어서 해줘
```

## Part 4: 프롬프트 캐싱 최적화로 API 비용 절감

### 프롬프트 캐싱이란?

Claude Code는 매번 AI에게 **전체 대화 컨텍스트**를 보냅니다. 프롬프트 캐싱은 이전에 보낸 동일한 내용을 재사용하여 **비용을 최대 90%까지 줄이는** 기능입니다.

```
일반 요청:    [전체 컨텍스트] → 100% 비용
캐시 히트:    [캐시된 부분 재사용] + [새 부분만 전송] → 10% 비용
```

### 최적화 1: CLAUDE.md 순서 배치

캐싱은 **앞에서부터 일치하는 부분**만 재사용합니다. 따라서 `CLAUDE.md`의 순서가 비용에 직접 영향을 줍니다.

**순서 원칙:**

```
상단: 프로젝트 규칙, 코딩 컨벤션     ← 거의 안 바뀜 (캐시 히트율 높음)
중단: 아키텍처, 기술 스택 정보        ← 가끔 바뀜
하단: 현재 작업, 진행 상황            ← 자주 바뀜 (캐시 미스 허용)
```

### 최적화 2: .claudeignore로 불필요한 파일 차단

**`.claudeignore` 추천 설정:**

```
# 의존성
node_modules/
vendor/
.venv/

# 빌드 산출물
dist/
build/
.next/

# 락 파일
package-lock.json
yarn.lock
pnpm-lock.yaml

# 대용량/바이너리
*.min.js
*.min.css
*.map
```

> 📊 이것만으로도 파일 읽기 시 **토큰 소비가 60~90% 줄어든다**는 테스트 결과가 있습니다.

### 최적화 3: 세션 쪼개지 않기

캐시는 **세션 내에서만 유효**합니다. 같은 10만 토큰짜리 파일을 읽고 질문 5개를 할 때:

| 방식 | 비용 배수 |
|------|-----------|
| 한 세션에서 연속 질문 | **~1.6배** |
| 매번 새 세션에서 질문 | **~6.2배** |

**캐시 유효 시간:**

| 플랜 | 캐시 유지 시간 |
|------|---------------|
| Max 플랜 | **1시간** |
| Pro 플랜 | **5분** |

**실전 규칙:**

```
✅ 관련된 작업은 하나의 세션에서 몰아서 처리
✅ 주제가 완전히 바뀔 때만 /clear로 리셋
❌ 질문마다 새 세션 열기
❌ 습관적으로 /clear 남발
```
