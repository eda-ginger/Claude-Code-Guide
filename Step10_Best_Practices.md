# Step 10: Claude Code 모범 사례

이 문서에서는 Claude Code를 최대한 효과적으로 활용하기 위한 **작업 전략, 세션 관리, 자동화 패턴, 흔한 실수 회피법**을 안내합니다.

---

## 핵심 원칙

Claude Code는 단순한 챗봇이 아니라 **에이전트 코딩 환경**입니다. 파일을 읽고, 명령을 실행하고, 변경을 수행하며, 자율적으로 문제를 해결합니다.

대부분의 모범 사례는 하나의 제약 조건에서 출발합니다:

> **컨텍스트 윈도우가 빠르게 채워지고, 채워질수록 성능이 저하됩니다.**

컨텍스트 윈도우는 모든 메시지, Claude가 읽은 파일, 명령 출력을 포함합니다. 단일 디버깅 세션만으로도 수만 개의 토큰을 소비할 수 있습니다. 컨텍스트가 거의 가득 차면 Claude는 이전 지시사항을 "잊거나" 실수가 늘어납니다.

**컨텍스트 윈도우는 관리해야 할 가장 중요한 리소스입니다.**

---

## 1. Claude에게 작업을 검증할 방법 제공하기

> Claude가 자신의 작업을 확인할 수 있도록 테스트, 스크린샷 또는 예상 출력을 포함하세요. **이것이 할 수 있는 가장 영향력 있는 단일 작업입니다.**

검증 수단 없이 Claude를 쓰면 그럴듯하지만 실제로는 작동하지 않는 코드가 나올 수 있습니다.

| 전략 | 비효율적 | 효과적 |
|------|----------|--------|
| **검증 기준 제공** | "이메일 주소를 검증하는 함수를 구현하세요" | "validateEmail 함수를 작성하세요. 테스트: `user@example.com`은 true, `invalid`는 false, `user@.com`은 false. 구현 후 테스트를 실행하세요" |
| **UI 시각적 검증** | "대시보드를 더 좋게 보이게 하세요" | "[스크린샷 붙여넣기] 이 디자인을 구현하세요. 결과의 스크린샷을 찍고 원본과 비교하세요" |
| **근본 원인 해결** | "빌드가 실패하고 있습니다" | "빌드가 이 오류로 실패합니다: [오류]. 수정하고 빌드가 성공하는지 확인하세요. 오류를 억제하지 마세요" |

## 2. 먼저 탐색하고, 계획하고, 그 다음 코드 작성하기

> 연구 및 계획을 구현과 분리하여 잘못된 문제를 해결하는 것을 피하세요.

Claude가 바로 코딩으로 뛰어들면 잘못된 문제를 해결할 수 있습니다. **Plan Mode**(`Shift+Tab`으로 전환)를 사용하여 탐색을 실행과 분리하세요.

### 권장 워크플로우: 4단계

**1단계: 탐색** (Plan Mode)
```
/src/auth를 읽고 세션 및 로그인을 어떻게 처리하는지 이해하세요.
```

**2단계: 계획** (Plan Mode)
```
Google OAuth를 추가하고 싶습니다. 어떤 파일을 변경해야 합니까? 계획을 작성하세요.
```

**3단계: 구현** (Normal Mode로 전환)
```
계획에서 OAuth 흐름을 구현하세요. 콜백 핸들러에 대한 테스트를 작성하고,
테스트를 실행하고 실패를 수정하세요.
```

**4단계: 커밋**
```
설명적인 메시지로 커밋하고 PR을 열기
```

> 범위가 명확하고 수정이 작은 작업(오타 수정, 로그 추가 등)에는 계획이 필요 없습니다. diff를 한 문장으로 설명할 수 있다면 바로 실행하세요.

## 3. 프롬프트에서 구체적인 컨텍스트 제공하기

> 지시사항이 정확할수록 필요한 수정이 적습니다.

| 전략 | 비효율적 | 효과적 |
|------|----------|--------|
| **작업 범위 지정** | "foo.py에 대한 테스트 추가" | "사용자가 로그아웃된 경우의 엣지 케이스를 다루는 foo.py 테스트를 작성하세요. 모의 객체를 피하세요" |
| **소스 지적** | "ExecutionFactory가 왜 이상한 API를 가지고 있습니까?" | "ExecutionFactory의 git 히스토리를 살펴보고 API가 어떻게 되었는지 요약하세요" |
| **기존 패턴 참조** | "캘린더 위젯 추가" | "HotDogWidget.php 패턴을 따라 캘린더 위젯을 구현하세요. 기존 라이브러리만 사용하세요" |
| **증상 설명** | "로그인 버그 수정" | "세션 시간 초과 후 로그인이 실패합니다. src/auth/의 토큰 새로 고침을 확인하세요. 실패 테스트를 작성한 다음 수정하세요" |

### 풍부한 콘텐츠 제공하기

- **`@`로 파일 참조**: 코드 위치를 설명하는 대신 직접 참조
- **이미지 붙여넣기**: 프롬프트에 복사/붙여넣기 또는 드래그 앤 드롭
- **URL 제공**: 문서 및 API 참조 링크 전달
- **데이터 파이프**: `cat error.log | claude`로 파일 내용 직접 전송

## 4. 효과적인 CLAUDE.md 작성하기

> `/init`을 실행하여 시작 CLAUDE.md를 생성한 다음 시간이 지남에 따라 개선하세요.

CLAUDE.md는 매 대화 시작 시 자동으로 로드됩니다. 짧고 핵심적으로 유지하세요.

```markdown
# 코드 스타일
- ES 모듈(import/export) 구문 사용, CommonJS(require) 사용 금지
- 가능하면 import를 구조 분해 (예: import { foo } from 'bar')

# 워크플로우
- 코드 변경 완료 시 타입 체크 확인
- 전체 테스트 스위트가 아닌 단일 테스트 실행 선호
```

### 포함 vs 제외

| 포함할 것 | 제외할 것 |
|-----------|-----------|
| Claude가 추측할 수 없는 Bash 명령 | Claude가 코드를 읽어서 파악할 수 있는 것 |
| 기본값과 다른 코드 스타일 규칙 | Claude가 이미 아는 표준 언어 규칙 |
| 테스트 지시사항 및 선호하는 러너 | 상세한 API 문서 (대신 링크) |
| 저장소 에티켓 (분기 이름, PR 규칙) | 자주 변경되는 정보 |
| 일반적인 함정 또는 명백하지 않은 동작 | 파일별 코드베이스 설명 |

> 각 줄에 대해 물어보세요: *"이것을 제거하면 Claude가 실수를 할까?"* 그렇지 않으면 삭제하세요. 부풀려진 CLAUDE.md는 Claude가 중요한 규칙을 무시하게 만듭니다.

### CLAUDE.md 위치별 적용 범위

| 위치 | 적용 범위 |
|------|-----------|
| `~/.claude/CLAUDE.md` | 모든 프로젝트에 공통 적용 |
| `./CLAUDE.md` | 해당 프로젝트 전체 (Git 커밋 대상) |
| `./하위디렉토리/CLAUDE.md` | 해당 디렉토리 작업 시 추가 적용 |

## 5. 세션 관리하기

### 조기에 자주 방향 수정하기

최고의 결과는 긴밀한 피드백 루프에서 나옵니다.

| 단축키 | 기능 |
|--------|------|
| `Esc` | Claude의 작업을 중간에 중지 (컨텍스트 보존) |
| `Esc + Esc` 또는 `/rewind` | 이전 대화 및 코드 상태 복원 |
| `"Undo that"` | Claude에게 변경 사항 되돌리기 요청 |
| `/clear` | 컨텍스트 완전 초기화 |

> 한 세션에서 같은 문제에 대해 **두 번 이상 수정**했다면, `/clear`를 실행하고 배운 내용을 통합한 더 나은 프롬프트로 새로 시작하세요. 누적된 수정이 있는 긴 세션보다 더 나은 프롬프트가 있는 깨끗한 세션이 **거의 항상** 더 잘 작동합니다.

### 컨텍스트 적극적으로 관리하기

| 방법 | 용도 |
|------|------|
| `/clear` | 관련 없는 작업 간에 컨텍스트 완전 재설정 |
| `/compact <지시>` | 특정 내용을 유지하며 압축 (예: `/compact API 변경 내용만 유지`) |
| `/rewind` → **Summarize** | 특정 지점부터만 압축하고 이전 컨텍스트 유지 |
| Subagent 활용 | 탐색을 별도 컨텍스트에서 실행하여 메인 대화를 깨끗하게 유지 |

### Subagent로 조사하기

> 컨텍스트가 기본 제약 조건이므로 subagent는 가장 강력한 도구 중 하나입니다.

Claude가 코드베이스를 연구할 때 많은 파일을 읽으며 컨텍스트를 소비합니다. Subagent는 별도의 컨텍스트 윈도우에서 실행되고 요약만 보고합니다:

```
subagent를 사용하여 인증 시스템이 토큰 새로 고침을 어떻게 처리하는지,
그리고 재사용해야 할 기존 OAuth 유틸리티가 있는지 조사하세요.
```

구현 후 검증에도 사용할 수 있습니다:

```
subagent를 사용하여 이 코드를 엣지 케이스에 대해 검토하세요
```

### 대화 재개하기

```bash
claude --continue    # 가장 최근 대화 재개
claude --resume      # 최근 대화에서 선택
```

`/rename`으로 세션에 설명적인 이름을 지정하세요 (예: `"oauth-migration"`, `"debugging-memory-leak"`).

## 6. Claude가 당신을 인터뷰하도록 하기

더 큰 기능의 경우, 최소한의 프롬프트로 시작하고 Claude에게 인터뷰를 요청하세요:

```
[간단한 설명]을 빌드하고 싶습니다. AskUserQuestion 도구를 사용하여 자세히 인터뷰해주세요.

기술 구현, UI/UX, 엣지 케이스, 트레이드오프에 대해 질문하세요.
명백한 질문을 하지 마세요, 제가 고려하지 않았을 어려운 부분을 파고드세요.

모든 것을 다룰 때까지 인터뷰를 계속한 다음 SPEC.md에 완전한 사양을 작성하세요.
```

사양이 완료되면 **새 세션**을 시작하여 구현하세요. 깨끗한 컨텍스트에서 작성된 사양을 참조하며 작업합니다.

## 7. 자동화 및 확장하기

### 비대화형 모드

```bash
# 일회성 쿼리
claude -p "이 프로젝트가 무엇을 하는지 설명하세요"

# 구조화된 출력
claude -p "모든 API 엔드포인트 나열" --output-format json

# 실시간 스트리밍
claude -p "이 로그 파일 분석" --output-format stream-json
```

### 여러 세션 병렬 실행

**Writer/Reviewer 패턴:**

| 세션 A (작성자) | 세션 B (검토자) |
|----------------|----------------|
| `속도 제한기 구현` | |
| | `rateLimiter.ts를 검토하세요. 엣지 케이스, 경쟁 조건을 찾으세요` |
| `검토 피드백을 반영하세요: [세션 B 출력]` | |

### 파일 전체에 Fan Out

대규모 마이그레이션이나 분석 시 병렬 호출로 작업을 분배합니다:

```bash
# 1. 마이그레이션 대상 파일 목록 생성
claude -p "마이그레이션이 필요한 모든 Python 파일 나열" > files.txt

# 2. 루프로 병렬 처리
for file in $(cat files.txt); do
  claude -p "React에서 Vue로 $file 마이그레이션. OK 또는 FAIL 반환." \
    --allowedTools "Edit,Bash(git commit *)"
done
```

> 처음 2~3개 파일에서 테스트하고, 프롬프트를 개선한 다음 전체 집합에서 실행하세요.

## 8. 흔한 실패 패턴 피하기

| 실패 패턴 | 증상 | 해결 방법 |
|-----------|------|-----------|
| **주방 싱크 세션** | 한 세션에서 관련 없는 여러 작업을 섞음. 컨텍스트가 관련 없는 정보로 가득 참 | 관련 없는 작업 간에 `/clear` |
| **반복적 수정** | Claude가 잘못하고, 수정하고, 또 잘못하고... 컨텍스트가 실패한 접근 방식으로 오염 | 두 번 실패 후 `/clear`하고 더 나은 프롬프트로 재시작 |
| **과도한 CLAUDE.md** | CLAUDE.md가 너무 길어서 Claude가 중요한 규칙을 무시 | 무자비하게 정리. Claude가 이미 올바르게 하는 것은 삭제 |
| **신뢰-검증 간격** | 그럴듯하지만 엣지 케이스를 처리하지 않는 코드 생성 | 항상 검증(테스트, 스크립트, 스크린샷) 제공 |
| **무한 탐색** | 범위 없이 "조사하세요"라고 요청. Claude가 수백 개 파일을 읽으며 컨텍스트 고갈 | 좁게 범위 지정하거나 subagent 사용 |

## 9. 직관 개발하기

이 가이드의 패턴은 시작점이지 정답이 아닙니다.

- 때로는 컨텍스트가 누적되도록 **해야** 합니다 — 깊이 있는 기록이 가치 있을 때
- 때로는 계획을 건너뛰고 Claude가 파악하도록 **해야** 합니다 — 탐색적 작업일 때
- 때로는 모호한 프롬프트가 정확히 맞습니다 — Claude가 문제를 해석하는 방식을 보고 싶을 때

**작동하는 것에 주의를 기울이세요.** Claude가 훌륭한 출력을 생성할 때 당신이 한 것을 주목하세요: 프롬프트 구조, 제공한 컨텍스트, 사용한 모드. 어려움을 겪을 때는 왜인지 물어보세요:

- 컨텍스트가 너무 시끄러웠나?
- 프롬프트가 너무 모호했나?
- 작업이 한 번에 너무 컸나?

시간이 지남에 따라 **구체적일 때와 개방적일 때, 계획할 때와 탐색할 때, 컨텍스트를 지울 때와 누적할 때**를 아는 직관이 생깁니다. 어떤 가이드도 이 직관을 대체할 수 없습니다.

---
> 💡 **Tip:** 누적된 수정이 있는 긴 세션보다, 더 나은 프롬프트가 있는 깨끗한 세션이 거의 항상 더 나은 결과를 만듭니다. 막히면 `/clear`부터 하세요.
