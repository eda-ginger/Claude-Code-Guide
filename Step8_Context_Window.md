# Step 8: 컨텍스트 윈도우 이해하기

이 문서에서는 Claude의 **컨텍스트 윈도우가 작동하는 방식**과 이를 **효과적으로 관리하는 전략**을 안내합니다.

---

## 1. 컨텍스트 윈도우란?

"컨텍스트 윈도우"는 Claude가 응답을 생성할 때 참조할 수 있는 **모든 텍스트의 총량**입니다. 응답 자체도 포함됩니다.

쉽게 말해 Claude의 **"작업 메모리"**입니다.

- 컨텍스트 윈도우가 **크면** → 더 복잡하고 긴 대화를 처리할 수 있음
- 컨텍스트 윈도우가 **작으면** → 긴 대화에서 일관성 유지가 어려워짐

> 이것은 Claude가 훈련된 대규모 데이터와는 다릅니다. 훈련 데이터는 "장기 기억", 컨텍스트 윈도우는 "단기 작업 메모리"에 해당합니다.

## 2. 컨텍스트 윈도우의 작동 방식

### 2.1 토큰이 쌓이는 구조

대화가 진행될수록 토큰이 **선형적으로 축적**됩니다:

```
[턴 1] 사용자 메시지 + Claude 응답
[턴 2] 턴 1 전체 + 사용자 메시지 + Claude 응답
[턴 3] 턴 1~2 전체 + 사용자 메시지 + Claude 응답
  ...
→ 계속 누적되어 결국 한계에 도달
```

각 턴은 두 단계로 구성됩니다:
- **입력 단계**: 모든 이전 대화 기록 + 현재 사용자 메시지
- **출력 단계**: Claude의 텍스트 응답 생성 (이것이 다음 턴의 입력 일부가 됨)

### 2.2 모델별 컨텍스트 윈도우 크기

| 모델 | 기본 컨텍스트 윈도우 | 확장 컨텍스트 (베타) |
|------|---------------------|---------------------|
| Claude Opus 4.6 | 200K 토큰 | 1M 토큰 |
| Claude Sonnet 4.6 | 200K 토큰 | 1M 토큰 |
| Claude Sonnet 4.5 | 200K 토큰 | 1M 토큰 |
| Claude Sonnet 4 | 200K 토큰 | 1M 토큰 |
| Claude Haiku 4.5 | 200K 토큰 | - |

> 1M 토큰 컨텍스트는 사용 티어 4 이상 조직에서 베타로 사용 가능하며, 200K 초과 시 프리미엄 요금(입력 2배, 출력 1.5배)이 적용됩니다.

## 3. 확장된 사고(Extended Thinking)와 컨텍스트

Claude는 복잡한 문제를 풀 때 **확장된 사고(thinking)** 블록을 생성합니다. 이 사고 블록도 토큰을 소비하지만, 특별한 처리 방식이 있습니다.

### 핵심 포인트: 이전 사고 블록은 자동으로 제거됨

```
[턴 1] 입력 → 사고 블록 + 응답 출력
[턴 2] 입력(턴 1의 사고 블록은 제외됨) → 새로운 사고 블록 + 응답 출력
[턴 3] 입력(이전 사고 블록 모두 제외됨) → 새로운 사고 블록 + 응답 출력
```

- 사고 블록은 **생성 시점에만 한 번** 출력 토큰으로 청구됨
- **후속 턴에서는 자동으로 제거**되어 컨텍스트 공간을 절약
- 유효 계산: `컨텍스트 = (입력 토큰 - 이전 사고 토큰) + 현재 턴 토큰`

이 덕분에 Claude가 깊이 사고하더라도 다음 턴의 컨텍스트 공간을 낭비하지 않습니다.

### 도구 사용 중 사고 블록 처리

확장된 사고와 도구 사용을 함께 쓸 때는 주의할 점이 있습니다:

1. **도구 결과를 반환할 때**: 해당 도구 요청에 수반되는 사고 블록은 **반드시 함께 반환**해야 합니다
2. **도구 사용 주기 완료 후**: 이전 사고 블록은 다시 자동으로 제거됩니다
3. **사고 블록 수정 금지**: 암호화 서명으로 진위를 확인하므로, 수정 시 API 오류가 발생합니다

> Claude 4 모델은 **인터리브 사고**를 지원하여 도구 호출 사이사이에 사고하고, 도구 결과를 받은 후 추가 추론이 가능합니다.

## 4. 컨텍스트 인식 기능

Claude Sonnet 4.6, Sonnet 4.5, Haiku 4.5는 **컨텍스트 인식** 기능을 갖추고 있습니다. 남은 컨텍스트 공간을 실시간으로 추적합니다.

### 작동 방식

대화 시작 시 총 예산 정보를 받습니다:

```xml
<budget:token_budget>200000</budget:token_budget>
```

각 도구 호출 후 남은 용량 업데이트를 받습니다:

```xml
<system_warning>Token usage: 35000/200000; 165000 remaining</system_warning>
```

### 왜 유용한가?

시계 없이 요리 대회에 참가하는 것을 상상해보세요. 컨텍스트 인식이 없으면 Claude도 마찬가지입니다.

- 장시간 실행되는 에이전트 세션에서 **지속적인 집중** 유지
- 복잡한 작업에서 **신중한 토큰 관리** 가능
- 남은 토큰 수를 추측하는 대신 **정확하게 파악**하여 끝까지 작업 지속

## 5. Claude Code에서의 컨텍스트 관리

Claude Code를 사용할 때 컨텍스트 윈도우를 효과적으로 관리하는 실전 전략입니다.

### 5.1 `/compact` — 대화 압축

대화가 길어지면 컨텍스트를 압축합니다:

```
> /compact
```

특정 내용만 유지하도록 지정할 수도 있습니다:

```
> /compact 인증 관련 작업 내용만 유지해줘
```

### 5.2 `/clear` — 컨텍스트 초기화

완전히 다른 주제로 전환할 때 사용합니다:

```
> /clear
```

### 5.3 `/cost` — 토큰 사용량 확인

현재 세션의 토큰 사용량과 비용을 확인합니다:

```
> /cost
```

### 5.4 자동 압축

Claude Code는 컨텍스트 한계에 가까워지면 **자동으로 이전 메시지를 압축**합니다. 수동 개입 없이도 대화를 이어갈 수 있습니다.

### 5.5 CLAUDE.md 활용

반복적으로 설명하는 프로젝트 정보는 `CLAUDE.md`에 기록하면, 매 대화마다 토큰을 소비하지 않고도 Claude가 프로젝트 컨텍스트를 이해합니다.

## 6. 컨텍스트 윈도우 한계 도달 시

최신 Claude 모델(Claude Sonnet 3.7부터)은 컨텍스트 윈도우를 초과하면 자동 잘라내기 대신 **유효성 검사 오류를 반환**합니다. 이는 더 예측 가능한 동작을 제공하지만, 신중한 토큰 관리가 필요합니다.

### 대응 방법

| 상황 | 해결 방법 |
|------|-----------|
| 대화가 너무 길어짐 | `/compact`로 압축 또는 `/clear` 후 재시작 |
| 큰 파일을 통째로 읽음 | 필요한 부분만 선택적으로 읽기 |
| 반복적인 프로젝트 설명 | CLAUDE.md에 기록하여 자동 로드 |
| 에이전트 팀 사용 시 | 각 팀원이 독립 컨텍스트를 가지므로 작업 분배로 해결 |

## 7. 서버 측 압축 (API 개발자용)

API를 직접 사용하는 개발자의 경우, **서버 측 압축(compaction)**을 활용할 수 있습니다. 대화의 이전 부분을 자동으로 요약하여 컨텍스트 한계를 넘어서는 장시간 대화가 가능합니다.

더 세밀한 제어가 필요한 경우 **컨텍스트 편집**을 통해 다음 전략을 사용할 수 있습니다:
- **도구 결과 지우기**: 에이전트 워크플로에서 오래된 도구 결과 제거
- **사고 블록 지우기**: 확장된 사고와 함께 사고 블록 관리

## 8. 핵심 요약

| 개념 | 설명 |
|------|------|
| **컨텍스트 윈도우** | Claude의 작업 메모리. 대화가 길어지면 선형적으로 채워짐 |
| **확장된 사고** | 생성 시 한 번만 청구, 이후 턴에서 자동 제거 |
| **컨텍스트 인식** | 남은 토큰을 실시간 추적 (Sonnet 4.6, Sonnet 4.5, Haiku 4.5) |
| **압축** | `/compact`로 대화를 요약하여 공간 확보 |
| **1M 컨텍스트** | 베타 기능, 200K 초과 시 프리미엄 요금 적용 |

---
> 💡 **Tip:** Claude Code에서 작업할 때, 하나의 긴 대화로 모든 것을 처리하려 하지 마세요. 주제가 바뀌면 `/clear`로 새로 시작하고, 작업이 길어지면 `/compact`로 압축하는 습관을 들이면 컨텍스트 윈도우를 훨씬 효율적으로 활용할 수 있습니다.
